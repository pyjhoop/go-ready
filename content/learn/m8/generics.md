Go는 10년 넘게 제네릭 없이 버티다 1.18(2022)에서야 도입했다. 이 지각이 오히려 설계를 말해준다 — Go 팀은 "정말 필요한 최소한"만 넣으려 오래 고민했다. 그래서 제네릭을 **배우는 것**보다 **언제 쓰지 않을지**를 아는 게 더 관용적이다.
{.lead}

## 타입 매개변수와 제약

제네릭 이전엔 "어떤 타입이든 담는 슬라이스에서 최댓값 찾기" 같은 함수를 타입마다 복붙하거나 `any` + 타입 단언으로 우회해야 했다. 이제는 **타입 매개변수** `[T ...]`로 한 번만 쓴다.

{{< gocode file="main.go" output="9 z" >}}
```go
package main

import "fmt"

// T는 cmp.Ordered를 만족하는 어떤 타입 — 비교 가능한 것만 허용
func Max[T interface{ ~int | ~float64 | ~string }](s []T) T {
    m := s[0]
    for _, v := range s[1:] {
        if v > m { // 제약이 > 를 보장하므로 컴파일된다
            m = v
        }
    }
    return m
}

func main() {
    // 두 결과를 한 줄에 출력 → "9 z"
    fmt.Println(Max([]int{3, 9, 4}), Max([]string{"a", "z", "m"}))
}
```
{{< /gocode >}}

`[T interface{ ~int | ~float64 | ~string }]`가 **제약(constraint)**이다. T가 될 수 있는 타입의 집합을 정의한다. `>` 연산을 쓰려면 T가 순서 비교 가능한 타입임을 컴파일러가 알아야 하므로, 제약으로 그 후보를 좁힌다. 호출부는 `Max([]int{...})`처럼 타입을 **명시하지 않아도** 인자에서 추론된다.

## 제약은 인터페이스의 확장이다

Go는 제네릭을 위해 인터페이스 개념을 넓혔다. 기존 인터페이스가 "이런 메서드를 가진 타입"을 뜻했다면, 이제 **타입 집합**(`~int | ~string`)도 인터페이스로 표현한다. `~int`의 물결표(`~`)는 "int, 그리고 int를 기반으로 정의된 모든 타입"을 뜻한다. 표준 라이브러리는 흔한 제약을 미리 제공한다:

- `cmp.Ordered` — `<`, `>` 비교가 되는 모든 타입.
- `any` — 모든 타입(빈 인터페이스의 별칭).
- `constraints` 패키지(`golang.org/x/exp`) — `Integer`, `Float` 등.

그리고 `slices`·`maps` 패키지가 이 제네릭 위에 세워져 있다: `slices.Sort`, `slices.Contains`, `slices.Max`, `maps.Keys` 등을 타입 안전하게 쓴다.

{{< callout tip >}}
★ **인터페이스로 충분하면 제네릭을 쓰지 마라.** 이건 Go 커뮤니티의 강한 합의다. 제네릭이 진짜 값을 하는 곳은 **자료구조**(타입 안전한 스택·트리·캐시)와 **컬렉션 알고리즘**(Map/Filter/Max처럼 원소 타입에 무관한 연산)뿐이다. "다형적 동작"이 필요하면 여전히 인터페이스가 정답이다 — `io.Reader`를 제네릭으로 바꾸면 더 나빠진다. 판단 기준: 함수가 타입의 **행동(메서드)**에 의존하면 인터페이스, **구조(원소를 담고 나른다)**에만 의존하면 제네릭.
{{< /callout >}}

{{< callout warn >}}
제네릭은 강력해서 **과용**하기 쉽다. 타입 매개변수가 서너 개씩 붙고 제약이 중첩되기 시작하면, 읽는 사람은 구체 타입 두세 벌을 그냥 쓴 코드보다 훨씬 힘들어한다. "중복이 딱 두 번"이라면 대개 복붙이 제네릭보다 낫다. Go의 미덕은 표현력이 아니라 **6개월 뒤 남이 읽을 수 있음**이라는 걸 기억하라(모듈 1).
{{< /callout >}}
