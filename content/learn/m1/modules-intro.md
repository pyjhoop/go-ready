Go 코드를 처음 보면 `import "github.com/user/repo/pkg"`처럼 import 경로가 URL을 닮아 이상하게 느껴진다. 여기엔 이름 충돌을 원천 차단하는 설계가 있다.
{.lead}

## import 경로가 URL처럼 생긴 이유

Node의 npm이나 Java의 Maven Central은 **중앙 레지스트리**에 이름을 등록한다. 그래서 좋은 이름을 먼저 차지하려는 경쟁이 생기고, `left-pad` 같은 짧은 이름을 두고 전쟁이 벌어진다.

Go는 이 문제를 다르게 푼다. **패키지의 이름이 곧 위치**다. `github.com/user/repo/pkg`는 이름이자 소스가 있는 주소다. 중앙 레지스트리가 없으니 이름 선점 전쟁도 없다 — 네 저장소 주소가 곧 네 패키지의 유일한 이름이다.

- **패키지 = 디렉터리** — 한 디렉터리 안의 `.go` 파일들이 하나의 패키지를 이룬다.
- **`go.mod`** — 프로젝트 루트에 놓여 모듈의 이름(= import 경로의 접두사)과 Go 버전, 의존성을 정의한다.

{{< contrast from="Java / Node" note="중앙 레지스트리에 이름을 등록 → 이름 선점 경쟁이 생긴다" to="Go" goNote="go.mod + import 경로(URL) → 이름과 위치가 하나라 충돌이 없다" >}}

## 순환 import는 애초에 불가능하다

Go는 패키지 A가 B를 import하고 B가 다시 A를 import하는 **순환 의존을 컴파일 단계에서 금지**한다. 다른 언어라면 "가능은 하지만 피하세요" 정도의 조언이지만, Go에서는 그냥 빌드가 실패한다.

이건 불편이 아니라 **강제된 규율**이다. 의존성 그래프가 순환 없이 항상 트리(정확히는 DAG) 형태로 유지되면, 각 패키지의 책임 경계가 뚜렷해지고 빌드도 빨라진다. 순환이 생겼다는 건 대개 "이 두 패키지는 사실 하나이거나, 공통 부분을 세 번째 패키지로 빼야 한다"는 설계 신호다.

{{< callout warn >}}
**순환 import는 컴파일 에러다.** 에러가 나면 억지로 우회하지 말고 의존 방향을 다시 보라 — 공통 타입을 별도 패키지로 분리하거나, 인터페이스로 의존을 뒤집는 게 정석이다(모듈 4에서 다룬다).
{{< /callout >}}
