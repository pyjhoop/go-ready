다른 언어를 배울 때 가장 지치는 부분은 언어가 아니라 **주변 도구**다. 빌드 도구, 포매터, 린터, 패키지 매니저를 따로 고르고 설정한다. Go는 이걸 통째로 없앴다.
{.lead}

## 도구가 흩어지지 않는다는 것

Node를 시작하면 npm/yarn/pnpm, webpack/vite, prettier, eslint를 각각 고르고 설정 파일을 만든다. Java는 Maven이냐 Gradle이냐부터 정한다. 이 파편화는 단순한 번거로움이 아니라, **팀마다 환경이 달라지는** 원인이다.

Go는 `go` 하나에 필요한 걸 다 넣었다. 이건 1.1에서 본 "뺄셈의 설계"의 연장이다 — 선택지를 줄여서 팀 전체의 인지 비용을 낮춘다.

| 커맨드 | 언제 쓰나 |
|---|---|
| `go run` | 컴파일하고 바로 실행 (스크립트처럼 빠르게 확인) |
| `go build` | 실행 파일 생성 (배포용 단일 바이너리) |
| `go test` | 테스트 실행 (테스트 러너가 언어에 내장) |
| `go vet` | 흔한 실수 정적 분석 |
| `go fmt` | 표준 스타일로 코드 정렬 |
| `go mod` | 의존성·모듈 관리 |

## 가장 작은 프로그램 실행하기

`go run`은 컴파일 단계를 의식하지 않게 해준다. 파일 하나를 저장하고 바로 돌려볼 수 있다.

{{< gocode file="main.go" output="Hello, Go!" >}}
```go
package main

import "fmt"

func main() {
    fmt.Println("Hello, Go!")
}
```
{{< /gocode >}}

`go run main.go`는 내부적으로 컴파일해 임시 바이너리를 만들고 실행한 뒤 지운다. 배포할 결과물이 필요하면 `go build`로 진짜 실행 파일을 뽑는다 — 런타임도, 의존성 폴더도 필요 없는 **단일 바이너리** 하나가 나온다.

{{< contrast from="Node / Java" note="빌드·포맷·린트·패키지 도구를 따로 고르고 설정 파일로 조립한다" to="Go" goNote="go 하나로 run·build·test·fmt·mod가 전부 해결된다" >}}

## gofmt — 포매터의 탈을 쓴 사회 공학

`gofmt`(그리고 `go fmt`)는 코드 스타일을 **협상 불가능한 규칙**으로 못 박는다. 들여쓰기 탭이냐 스페이스냐, 중괄호를 어디 두냐 같은 논쟁이 Go 커뮤니티에는 없다. 답이 하나로 정해져 있기 때문이다.

{{< callout tip >}}
`gofmt`는 단순한 포매터가 아니라 **문화적 장치**다. 스타일 리뷰가 사라지면 리뷰어는 "동작이 맞는가"에만 집중할 수 있다. 코드가 전부 똑같이 생겼다는 건 남의 코드를 읽는 마찰이 사라진다는 뜻이다.
{{< /callout >}}

{{< callout info >}}
Go 프로젝트에는 설정 파일이 거의 없다. "설정할 게 없다"는 결핍이 아니라 **기능**이다 — 모두가 같은 규칙 위에서 움직인다.
{{< /callout >}}
