C를 아는 사람은 "포인터"라는 말에 긴장한다. Go의 포인터는 다르다 — **주소를 가리키긴 하지만, 산술은 못 한다.** 이 하나의 제약이 C 포인터의 공포 대부분을 걷어낸다.
{.lead}

## & 와 * — 그게 전부다

Go 포인터의 문법은 두 개뿐이다. `&`로 주소를 얻고, `*`로 그 주소의 값을 읽는다.

```go
x := 42
p := &x        // p는 x의 주소를 가리키는 *int
fmt.Println(*p) // 42 — 역참조(dereference)
*p = 100        // x를 100으로 바꾼다
```

`nil`은 포인터의 zero value이고, "아무것도 안 가리킴"을 뜻한다. nil 포인터를 역참조하면 panic이다.

## 포인터 산술이 없다는 것

C에서는 `p + 1`, `p++`로 포인터를 옮겨 다니며 메모리를 훑을 수 있다. 이 능력이 버퍼 오버런, 임의 메모리 접근 같은 재앙의 근원이었다. **Go는 포인터 산술을 아예 없앴다.** 포인터로 할 수 있는 건 "가리키는 값 읽기/쓰기"뿐, 주소를 더하거나 빼는 연산은 언어에 존재하지 않는다.

이 뺄셈 하나로 C 포인터 위험의 큰 축이 사라진다. 포인터는 이제 "이 값을 공유하겠다"는 안전한 참조일 뿐이다.

## 지역 변수 주소를 반환해도 안전하다

C 프로그래머가 가장 놀라는 지점이다. C에서 지역 변수의 주소를 반환하면, 함수가 끝나며 스택이 사라져 **댕글링 포인터**가 된다. Go에서는 이게 완전히 안전하고, 심지어 **관용적**이다.

```go
func newUser(name string) *User {
    return &User{Name: name} // 지역 값의 주소 반환 — 완전히 정상
}
```

어떻게 안전할까? Go 컴파일러의 **이스케이프 분석(escape analysis)**이 "이 값의 주소가 함수 밖으로 나가는구나"를 알아채고, 그 값을 스택 대신 **힙에 할당**한다. 그리고 아무도 안 가리키게 되면 **가비지 컬렉터(GC)**가 회수한다. 개발자는 이 과정을 신경 쓸 필요가 없다.

{{< callout tip >}}
Go 포인터가 안 무서운 이유는 **세 가지의 조합**이다: ① 포인터 산술 제거 ② 가비지 컬렉션 ③ 이스케이프 분석. 하나라도 빠지면 다시 무서워진다 — 산술이 있으면 메모리를 훑을 수 있고, GC가 없으면 언제 free할지 고민해야 하고, 이스케이프 분석이 없으면 지역 주소 반환이 위험해진다. Go는 셋을 다 갖춰서 "안전한 포인터"를 만들었다.
{{< /callout >}}

{{< contrast from="C (산술 O · 위험) / Java (포인터 숨김)" note="C는 강력하지만 위험하고, Java는 아예 감춰서 제어권을 뺏는다" to="Go" goNote="산술은 없애 안전을 얻고, & / * 로 참조는 명시적으로 제어한다" >}}

실무에서는 `new(T)`보다 `&T{...}`를 훨씬 자주 쓴다. `new(T)`는 zero value 포인터만 주지만, `&T{Field: v}`는 초기화와 주소 획득을 한 번에 하기 때문이다. `new`를 볼 일은 생각보다 드물다.
