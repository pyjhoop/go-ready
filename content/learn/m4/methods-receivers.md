Go의 메서드는 타입 바깥에 선언되고, 함수 이름 앞에 **리시버(receiver)**가 붙는다. 그리고 이 리시버가 값이냐 포인터냐에 따라, 메서드가 **복사본을 만지느냐 원본을 만지느냐**가 갈린다. 이게 이 모듈의 핵심 함정이다.
{.lead}

## 리시버 — 메서드가 붙는 대상

```go
type Counter struct{ n int }

func (c Counter) Value() int { return c.n } // c가 리시버
```

`(c Counter)`가 리시버다. 다른 언어의 `this`/`self`에 해당하지만, 결정적 차이가 있다 — 다른 언어에서 `this`는 **항상 참조**다. Go는 리시버를 **값으로 받을지 포인터로 받을지 직접 선택**한다. 이 선택이 메서드의 동작을 근본적으로 바꾼다.

## 값 리시버 vs 포인터 리시버

{{< diagram id="value-vs-pointer-receiver" caption="값 리시버는 복사본을 수정(원본 불변), 포인터 리시버는 원본을 직접 수정" >}}

{{< gocode file="main.go" output="after IncValue: 0 / after IncPtr:   1" >}}
```go
package main

import "fmt"

type Counter struct{ n int }

func (c Counter) IncValue() { c.n++ }  // 값 리시버 — 복사본을 수정
func (c *Counter) IncPtr()  { c.n++ }  // 포인터 리시버 — 원본을 수정

func main() {
    c := Counter{}
    c.IncValue()
    fmt.Print("after IncValue: ", c.n, " / ") // 0 — 안 바뀜!

    c.IncPtr()
    fmt.Print("after IncPtr:   ", c.n)         // 1 — 바뀜
}
```
{{< /gocode >}}

`IncValue`는 리시버 `c`의 **복사본**을 받는다. 그 복사본의 `n`을 아무리 올려도, 메서드가 끝나면 복사본은 버려지고 원본은 그대로다. `IncPtr`은 원본의 **주소**를 받으므로 원본을 직접 건드린다.

{{< callout warn >}}
★ **값 리시버는 복사본을 수정하므로 원본이 바뀌지 않는다.** 상태를 변경하는 메서드(setter, 카운터 증가 등)를 값 리시버로 만들면, 컴파일도 잘 되고 에러도 없지만 **아무 일도 일어나지 않는다.** "왜 값이 안 바뀌지?"의 90%는 값 리시버를 써야 할 곳에 포인터 리시버를 안 쓴 것이다.
{{< /callout >}}

## 실전 규칙 4가지

리시버 선택은 매번 고민할 문제가 아니다. 아래 규칙이면 거의 다 커버된다:

1. **상태를 변경하면 포인터** — 원본을 바꿔야 하는 메서드는 무조건 `*T`.
2. **큰 구조체는 포인터** — 값 리시버는 매 호출마다 통째로 복사한다. 필드가 많으면 복사 비용이 아깝다.
3. **한 타입 안에서는 일관되게** — 어떤 메서드는 값, 어떤 건 포인터로 섞지 마라. 하나라도 포인터가 필요하면 그 타입의 메서드 전부를 포인터로 통일하는 게 관용이다.
4. **슬라이스·맵 필드는 값 리시버여도 공유된다** — 리시버는 복사돼도 그 안의 슬라이스/맵 헤더가 가리키는 백킹 데이터는 같다(모듈 3의 그 원리). 그래서 값 리시버로도 슬라이스 원소는 바뀐다. 헷갈리기 쉬우니 의식하라.

{{< callout info >}}
포인터 리시버로 정의한 메서드는 **값의 메서드 셋(method set)에는 포함되지 않는다.** 이 미묘한 규칙이 다음다음 레슨(4.4 인터페이스)에서 "왜 이 타입이 인터페이스를 만족하지 않지?"라는 함정으로 되돌아온다. 지금은 "리시버 종류가 인터페이스 만족에도 영향을 준다"만 기억해두자.
{{< /callout >}}
