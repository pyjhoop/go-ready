다른 언어는 "문자열은 문자의 배열"이라고 가르친다. 편리한 거짓말이다. Go는 이 환상을 걷어내고 진실을 보여준다 — **문자열은 불변의 바이트열**이고, 그 바이트들은 UTF-8로 인코딩돼 있다.
{.lead}

## 문자열은 바이트다

Go에서 문자열은 읽기 전용 **byte 슬라이스**에 가깝다. `s[i]`는 i번째 **문자**가 아니라 i번째 **바이트**를 반환한다(타입도 `byte`, 즉 `uint8`).

ASCII만 쓰면 바이트 하나 = 문자 하나라 티가 안 난다. 하지만 한글·이모지·악센트 문자처럼 **여러 바이트로 인코딩되는 문자**가 들어오면 추상화가 무너진다.

{{< gocode file="main.go" output="len bytes = 6 / runes = 2" >}}
```go
package main

import "fmt"

func main() {
    s := "한글"
    fmt.Print("len bytes = ", len(s), " / ")   // 6 — 바이트 수!
    fmt.Print("runes = ", len([]rune(s)))       // 2 — 실제 글자 수
}
```
{{< /gocode >}}

`"한글"`은 눈에는 두 글자지만 `len(s)`는 **6**을 반환한다. UTF-8에서 한글 한 글자가 3바이트를 차지하기 때문이다. 실제 글자 수를 세려면 `[]rune`으로 변환해야 한다.

## byte 와 rune — 모듈 2의 복선 회수

모듈 2에서 `byte`와 `rune`이 별칭이라고만 하고 넘어갔다. 이제 그 쓸모가 드러난다:

- **`byte`** (= `uint8`) — 인코딩된 raw 바이트 하나. `s[i]`가 주는 것.
- **`rune`** (= `int32`) — 유니코드 코드 포인트 하나, 즉 사람이 "글자"라고 부르는 것.

`range`로 문자열을 순회하면 Go가 바이트를 **자동으로 rune 단위로 디코딩**해준다. 이때 인덱스는 바이트 위치라서 1씩이 아니라 **글자의 바이트 폭만큼 점프**한다.

```go
for i, r := range "a한" {
    fmt.Println(i, r)
}
// 0 97      ('a' — 1바이트)
// 1 54620   ('한' — 다음 인덱스가 2가 아니라 1에서 시작, 3바이트 차지)
```

인덱스가 0, 1로 이어지다가 그다음엔 4로 점프한다. `range`는 rune을 주지만 인덱스는 바이트 오프셋이라는, 처음엔 당황스러운 조합이다.

## 왜 이렇게 불편하게 만들었나

핵심 질문이다. 다른 언어는 왜 이런 고민을 안 시킬까? 답은 **비용을 숨겼을 뿐**이라는 것이다.

가변 길이 인코딩(UTF-8)에서 "n번째 글자"에 O(1)로 접근하는 것은 **원리적으로 불가능**하다. 각 글자의 바이트 폭이 제각각이라, n번째 글자를 찾으려면 앞에서부터 세어야 한다. 다른 언어들은 이 비용을 내부에 숨기거나(UTF-16 고정폭 가정 등), 잘못된 인덱싱으로 조용히 깨진 문자를 만들어낸다. Go는 숨기는 대신 **비용을 드러내서**, 개발자가 "지금 바이트를 세는가 글자를 세는가"를 의식하게 만든다.

{{< callout warn >}}
문자 수를 세거나 문자열을 자를 때 `len(s)`나 `s[i]`를 쓰면 멀티바이트 문자에서 깨진다. `len(s)`는 바이트 수이고, `s[i]`는 바이트 하나라 글자 중간을 잘라 깨진 문자를 만든다. 글자 단위로 다루려면 `[]rune(s)`로 변환하거나, `range`로 순회하거나, `utf8.RuneCountInString(s)`로 글자 수를 세라.
{{< /callout >}}
