이 프로젝트에서 가장 중요한 함정이다. 슬라이싱은 데이터를 복사하지 않는다 — 같은 백킹 배열을 들여다보는 **또 하나의 창**을 만들 뿐이다.
{.lead}

## 슬라이싱은 뷰(view)다

`s[1:3]` 같은 슬라이싱은 직관과 달리 **새 배열을 할당하지 않는다**. 앞 레슨에서 본 헤더만 새로 만들어, 원본과 **같은 백킹 배열**을 가리키게 한다.

```go
s := []int{10, 20, 30, 40}
sub := s[1:3]     // {20, 30} — 하지만 s와 같은 배열을 공유
sub[0] = 99       // s[1]도 함께 99로 바뀐다!
// s == {10, 99, 30, 40}
```

`sub`를 수정했는데 `s`가 바뀐다. 둘은 같은 메모리를 다른 창으로 보고 있기 때문이다. 여기까진 "공유되는구나" 정도로 넘어갈 수 있다. 진짜 함정은 `append`가 끼어들 때다.

## append의 재할당 규칙

`append`는 두 갈래로 갈린다. 이 분기가 슬라이스 버그의 핵심이다:

- **cap에 여유가 있으면** → 백킹 배열의 빈 자리에 그냥 써넣는다. **공유가 유지된다.**
- **cap을 초과하면** → 더 큰 배열을 새로 할당(대략 2배)하고 값을 복사한다. **이 순간 공유가 끊긴다.**

문제는 **재할당이 언제 일어나는지 예측하기 어렵다**는 것이다. `cap`은 눈에 잘 안 보이는 값이라, 어떤 append는 원본을 건드리고 어떤 append는 안 건드린다. 같은 코드가 데이터 크기에 따라 다르게 동작한다.

## 더 교묘한 버그 — 조용한 덮어쓰기

{{< gocode file="main.go" output="shared: [1 2 99] / safe:  [1 2 3]" >}}
```go
package main

import "fmt"

func main() {
    s := []int{1, 2, 3}

    // sub는 s와 백킹 배열을 공유하고, cap 여유가 있다
    sub := s[0:2]           // len=2, cap=3
    sub = append(sub, 99)   // 빈 자리에 써넣음 → s[2]를 덮어쓴다!
    fmt.Print("shared: ", s, " / ")

    // 방어: 3-인덱스 슬라이스로 cap을 잘라 강제 재할당
    t := []int{1, 2, 3}
    u := t[0:2:2]           // cap을 2로 제한
    u = append(u, 99)       // cap 초과 → 새 배열, t는 안전
    fmt.Print("safe:  ", t)
}
```
{{< /gocode >}}

첫 번째 `append(sub, 99)`는 `sub`의 cap에 여유가 있어서 **`s[2]`를 조용히 `99`로 덮어썼다.** 새 슬라이스에 값을 추가했을 뿐인데 원본이 오염됐다. 눈에 보이는 에러도 없이 데이터가 바뀐다.

## 방어법 — copy 와 full-slice 표현식

두 가지 무기가 있다:

- **`copy(dst, src)`** — 아예 독립된 배열로 복제한다. 공유가 필요 없으면 이게 가장 안전하다.
- **full-slice 표현식 `s[low:high:max]`** — 세 번째 인덱스로 `cap`을 명시적으로 제한한다. `t[0:2:2]`는 cap을 2로 잘라서, 다음 append가 반드시 새 배열을 할당하게 만든다.

{{< callout warn >}}
★ **라이브러리가 인자로 받은 슬라이스에 append하면 호출자의 데이터를 오염시킬 수 있다.** cap 여유가 있으면 호출자가 여전히 보고 있는 백킹 배열을 덮어쓰기 때문이다. 남에게 넘길 슬라이스를 append로 확장할 계획이면, 먼저 `copy`로 복제하거나 `s[low:high:high]`로 cap을 잘라 호출자와의 공유를 끊어라.
{{< /callout >}}

규칙으로 요약하면: **"이 슬라이스를 나 말고 누가 더 보고 있는가?"**를 항상 자문하라. 답이 "그렇다"이고 append를 할 거라면, 먼저 공유를 끊어야 한다.
