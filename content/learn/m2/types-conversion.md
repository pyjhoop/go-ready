Go는 `int`과 `float64`를 그냥 더하게 놔두지 않는다. 컴파일러가 멈춰 서서 "직접 변환하라"고 요구한다. 이 시끄러움이 바로 설계의 핵심이다.
{.lead}

## 왜 자동 변환을 막았나

C나 JavaScript는 숫자 타입을 알아서 섞어 계산한다. 편해 보이지만, 이 편의에는 **보이지 않는 청구서**가 붙는다:

- **조용한 정밀도 손실** — `int64`를 `float64`에 넣으면 큰 값에서 하위 비트가 날아간다.
- **부호 꼬임** — `uint`와 `int`가 섞이면 음수가 거대한 양수로 뒤집힌다.
- **안 보이는 오버플로** — 좁은 타입으로 자동 축소되며 값이 잘린다.

이런 버그의 공통점은 **컴파일도 테스트도 통과한다**는 것이다. 경계값이 들어오는 프로덕션에서야 터진다. Go는 이 부류의 버그를 언어 차원에서 없애기 위해, 타입이 다른 값끼리의 연산을 **컴파일 에러**로 만든다.

{{< gocode file="main.go" output="7.14" >}}
```go
package main

import "fmt"

func main() {
    i := 4       // int
    f := 3.14    // float64

    // fmt.Println(i + f)  // ← 컴파일 에러: mismatched types int and float64

    sum := float64(i) + f  // 의도를 명시적으로 표기
    fmt.Println(sum)
}
```
{{< /gocode >}}

`float64(i)`는 "여기서 `i`를 부동소수점으로 다루겠다"는 **의사 표시**다. 변환 표기가 코드에 남으면, 나중에 읽는 사람이 "이 지점에서 값의 의미가 바뀐다"를 즉시 본다.

{{< contrast from="JS / Python" note="타입이 다른 숫자를 알아서 섞어 계산 → 정밀도 손실이 조용히 발생" to="Go" goNote="다른 타입끼리 연산은 컴파일 에러 → 변환을 항상 명시" >}}

## 리터럴은 유연하고, 변수는 엄격하다

여기서 초보자가 헷갈린다. `var x float64 = 3`은 되는데 왜 `float64(i) + f`는 변환이 필요할까?

답은 **리터럴(literal)과 타입이 정해진 변수는 다르다**는 것이다. `3`이나 `3.14` 같은 숫자 리터럴은 아직 타입이 확정되지 않은 "상수"라서, 문맥에 맞춰 유연하게 맞춰진다. 하지만 `i`처럼 이미 `int`으로 **타입이 박힌 변수**는 다른 타입 자리에 자동으로 들어가지 않는다.

{{< callout warn >}}
숫자 **리터럴**은 문맥에 맞게 유연하게 해석되지만(`var f float64 = 10`은 OK), 타입이 정해진 **변수**끼리는 반드시 명시적으로 변환해야 한다(`float64(i) + f`). "왜 어떤 건 되고 어떤 건 안 되지?"의 답은 거의 항상 이 리터럴 대 변수의 구분이다.
{{< /callout >}}

## byte 와 rune — 미리 깔아두는 복선

기본 타입 중 두 개는 이름이 특이하다. `byte`와 `rune`이다. 사실 이 둘은 새로운 타입이 아니라 **별칭(alias)**이다:

- `byte` = `uint8` (0~255, 즉 바이트 하나)
- `rune` = `int32` (유니코드 코드 포인트 하나)

지금은 "문자열을 다룰 때 다시 만난다" 정도만 기억해두면 된다. 왜 `char` 타입 대신 이 둘로 쪼갰는지는 모듈 3의 문자열·rune 레슨에서 본격적으로 풀린다 — Go가 "문자열은 문자의 배열"이라는 흔한 거짓말을 어떻게 걷어내는지가 거기 있다.
